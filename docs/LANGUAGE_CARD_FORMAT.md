# Language Card JSON Format

## Purpose

This file describes the JSON format for language cards used by Language Crossword Lab.

It is intended as an authoring guide. A learner may give this document to an external LLM or use it manually to create valid JSON. The app itself does not need to know whether cards were written by a person or generated by an LLM.

## External JSON authoring mode

Use this mode when writing cards manually or asking an external LLM to create a file for import.

Return only the JSON array described below. Do not use app tool calls, include internal ids, or add surrounding prose.

### Output Format

Return a JSON array of language card objects.

Do not wrap the JSON in Markdown.

Do not add comments inside the JSON.

Do not include explanatory text before or after the JSON.

## In-app tool-calling mode

Use this mode only inside the Language Lab AI Assistant.

Follow the same supported-language, card-quality, one-meaning-per-card, and duplicate rules in this guide. Use the four bounded read tools to inspect the current library. Never invent an id for an existing card or card set; obtain existing ids from read-tool results.

Only propose writes through `propose_library_operation`. That tool stages a plan for user review and does not apply changes. Do not claim that a proposed operation has already changed the library, and do not attempt direct writes, global card deletion, or card-set archive or deletion.

## Supported Languages

Use these language codes:

- `ru` for Russian;
- `en` for English;
- `es` for Spanish.

## Card Object

A language card represents one learning unit. It can be a single word, a phrase, or a short expression.

The app creates internal ids by itself. Do not add an `id` field unless a future version explicitly asks for it.

### Required Field

Each card must include `translations`.

`translations` must contain at least two supported language keys.

Each translation value must be a non-empty string.

Example:

```json
{
  "translations": {
    "ru": "аэропорт",
    "en": "airport",
    "es": "aeropuerto"
  }
}
```

### Optional Fields

Cards may include:

- `definitions`;
- `examples`;
- `tags`;
- `difficulty`.

## Translations

`translations` stores equivalent words or phrases across languages.

Values may be:

- a single word;
- a multi-word phrase;
- a short expression.

Good examples:

```json
{
  "translations": {
    "ru": "билет",
    "en": "ticket",
    "es": "billete"
  }
}
```

```json
{
  "translations": {
    "ru": "Я бы хотел билет",
    "en": "I would like a ticket",
    "es": "Quisiera un billete"
  }
}
```

Do not put unrelated meanings into one card.

Bad example:

```json
{
  "translations": {
    "ru": "ключ",
    "en": "key, spring, clue",
    "es": "llave"
  }
}
```

Instead, create separate cards for separate meanings.

## Definitions

`definitions` is optional.

Definitions must use the same language as their key.

Example:

```json
{
  "translations": {
    "ru": "аэропорт",
    "en": "airport",
    "es": "aeropuerto"
  },
  "definitions": {
    "en": "A place where airplanes take off, land, and passengers travel through.",
    "ru": "Место, где самолёты взлетают, садятся и принимают пассажиров.",
    "es": "Lugar donde los aviones despegan, aterrizan y reciben pasajeros."
  }
}
```

Definitions are hints. They are not required.

When the target language is English, the app uses only `definitions.en` as a definition hint. Definitions in other languages are not used as definition hints for that exercise.

## Examples

`examples` is optional.

Examples are useful for missing-word-in-sentence exercises.

Use language keys under `examples`. Each language key contains an array of example objects.

Each example object should contain:

- `sentence`: a full sentence;
- `answer`: the word or phrase that can be removed from the sentence.

Example:

```json
{
  "translations": {
    "ru": "Я бы хотел билет",
    "en": "I would like a ticket",
    "es": "Quisiera un billete"
  },
  "examples": {
    "en": [
      {
        "sentence": "I would like a ticket to Madrid.",
        "answer": "I would like a ticket"
      }
    ],
    "es": [
      {
        "sentence": "Quisiera un billete a Madrid.",
        "answer": "Quisiera un billete"
      }
    ]
  }
}
```

If examples are missing, the card can still be used in other exercises.

## Tags

`tags` is optional.

Use tags for topic, grammar, or learning context.

Examples:

```json
"tags": ["travel", "airport", "phrases"]
```

Tags should be short lowercase strings.

## Difficulty

`difficulty` is optional.

Recommended values:

- `easy`
- `medium`
- `hard`

Example:

```json
"difficulty": "easy"
```

## Complete Example

```json
[
  {
    "translations": {
      "ru": "аэропорт",
      "en": "airport",
      "es": "aeropuerto"
    },
    "definitions": {
      "en": "A place where airplanes take off, land, and passengers travel through."
    },
    "tags": ["travel"],
    "difficulty": "easy"
  },
  {
    "translations": {
      "ru": "Я бы хотел билет",
      "en": "I would like a ticket",
      "es": "Quisiera un billete"
    },
    "definitions": {
      "en": "A polite phrase used when buying a ticket."
    },
    "examples": {
      "en": [
        {
          "sentence": "I would like a ticket to Madrid.",
          "answer": "I would like a ticket"
        }
      ]
    },
    "tags": ["travel", "phrases"],
    "difficulty": "medium"
  }
]
```

## Duplicate Guidance

Avoid generating duplicate cards.

A card may be treated as a duplicate if any translation value matches an existing card in any supported language.

If a duplicate provides missing information, the app may safely merge it. If it conflicts with existing data, the app stores it as a pending duplicate for later review.

## Quality Rules

Follow these rules when creating cards:

- Use at least two translations per card.
- Prefer all three languages when possible.
- Keep one meaning per card.
- Do not put comma-separated alternatives into one translation field.
- Do not mix unrelated meanings.
- Keep definitions concise.
- Write definitions in the same language as the definition key.
- Use examples that naturally contain the target word or phrase.
- Keep tags short and useful.
- Return valid JSON only.
